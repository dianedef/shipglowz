le Reader ne devrait pas écrire la doc, même s’il la connaît le       mieux.                                                                                                                                                            Je garderais cette séparation :                                                                                                                                   - Reader : propriétaire du diagnostic documentaire.                                Il lit le corpus, comprend le code, maintient la carte code-docs, puis           produit un Documentation Update Plan.                                          - Sequential Executor ou Docs Executor : applique les modifications de doc.        Il reçoit les fichiers exacts, sections exactes, raisons, niveau de priorité,    et écrit.                                                                      - Reader : relit après coup.                                                       Il vérifie que la doc mise à jour correspond bien au changement de code et       qu’il ne manque pas de surface documentaire.                                                                                                                    Pourquoi je ne ferais pas écrire le Reader : son intérêt principal est d’être    stable, read-only, réutilisable et fiable. Si on lui donne le droit d’écrire,    il devient aussi une source potentielle de conflit, surtout entre waves.                                                                                          Le flow idéal :                                                                                                                                                   Fin de wave                                                                      -> Executors rendent leurs changements                                           -> Reader produit Documentation Update Plan                                      -> Integrator valide les docs à toucher                                          -> Sequential Executor applique la doc                                           -> Reader relit et confirme                                                      -> Wave suivante ou sf-end                                                                                                                                        Donc oui : le Reader peut mettre à jour le champ / plan technique docs dans son  rapport de fin de cycle, mais pas modifier les fichiers. La doc réelle devrait   être appliquée séquentiellement par un executor write-capable, puis relue par    le Reader
