# Bug Record: BUG-2026-05-03-001

## Summary

- Title: New ShipFlow skill is not visible in Codex because runtime symlinks are missing
- Current status: `closed`
- Severity: `medium`
- Owner: unassigned
- Next command: none

## Reproduction

- Preconditions:
  - A new ShipFlow skill exists under `skills/<name>/SKILL.md`.
  - The current user's Codex and Claude skill directories already contain symlinks for older ShipFlow skills.
- Steps:
  1. Create and ship `skills/sg-skill-build/SKILL.md`.
  2. Inspect `~/.codex/skills/` and `~/.claude/skills/`.
  3. Start or inspect a Codex session's available skills.

## Expected Behavior

- A newly created ShipFlow skill is visible to the current operator runtimes.
- `~/.codex/skills/<name>` and `~/.claude/skills/<name>` resolve to `${SHIPFLOW_ROOT:-$HOME/shipglowz}/skills/<name>` and expose `SKILL.md`.
- `sg-skill-build` enforces this runtime publication gate for future new or renamed skills.

## Observed Behavior

- `skills/sg-skill-build/SKILL.md` existed in the repository.
- `~/.codex/skills/sg-skill-build` and `~/.claude/skills/sg-skill-build` were missing, so the skill was not visible through the runtime skill discovery surface.
- `sg-skill-build` documented budget, docs, public content, verify, and ship gates, but did not include a runtime symlink gate.

## Evidence

- User report on 2026-05-03: the new master skill was not visible in Codex.
- Local listing of `~/.codex/skills` and `~/.claude/skills` showed many ShipFlow skill symlinks, but no `sg-skill-build` entry.
- `install.sh` has `configure_skills`, `ensure_skill_link`, and `verify_skill_link`, but the new skill lifecycle did not call or mirror that runtime publication step.
