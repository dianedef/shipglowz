# Bug Record: BUG-2026-05-19-001

## Summary

- Title: Terminal TUI project filter does not scope specs
- Current status: `fixed-pending-verify`
- Severity: `medium`
- Owner: Diane
- Next command: `/sg-test --retest BUG-2026-05-19-001`

## Reproduction

- Preconditions:
  - ShipFlow Terminal TUI has project and spec data loaded.
  - Specs include `project:` frontmatter across more than one project.
- Steps:
  1. Launch the TUI with `tui`.
  2. Type a full project name in the project panel.
  3. Observe the specs section.

## Expected Behavior

- When a project filter is active, the specs section should only show specs matching the selected/matching project or the typed project scope.
- If no spec belongs to that project, the specs section should say no specs match.

## Observed Behavior

- The project list was filtered, but specs stayed global.
- This made specs from other projects appear even after typing a full project name.

## Evidence

- Root cause found in `tui/src/viewModels/dashboard.ts`: `filteredSpecs` only used `specFilter` and ignored `projectFilter`.
- Fixed with project-aware spec scoping and `project:` parsing from spec frontmatter.
- Second root cause found from `p:"gocharb"` evidence: full project paths contributed generic aliases such as `home` and `claude`, which matched every spec path under `/home/claude/...`.
- Fixed by removing full-path segments from project aliases and preferring spec `project:` metadata for scoped matching.
- Validation:
  - `PATH=/home/claude/.bun/bin:$PATH bun run typecheck`
  - `PATH=/home/claude/.bun/bin:$PATH bun test`
  - `git diff --check`
  - Node smoke on real data: `contentflow_app` excludes the ShipFlow Terminal TUI spec and shows ContentFlow specs.
  - Node smoke on real data: `gocharb` with `gocharbon_quiz` selected excludes ShipFlow specs and returns `No specs match`.
