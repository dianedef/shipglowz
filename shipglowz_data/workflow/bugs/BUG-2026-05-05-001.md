# Bug Record: BUG-2026-05-05-001

## Summary

- Title: ShipFlow tracking init should not create project TASKS.md symlinks
- Current status: `fix-attempted`
- Severity: `medium`
- Owner: unassigned
- Next command: `/sg-test --retest BUG-2026-05-05-001`

## Reproduction

- Preconditions:
  - A project has a `TASKS.md` symlink that points to an old or missing ShipFlow data path.
  - The symlink path itself still exists in the project directory, but its target does not.
- Steps:
  1. Start or restart the project through ShipFlow.
  2. Let ShipFlow initialize project tracking after PM2 startup.
  3. Observe tracking initialization output.

## Expected Behavior

- ShipFlow should keep its operational project tracker at `${SHIPFLOW_DATA_DIR:-$HOME/shipglowz_data}/projects/<project>/TASKS.md`.
- ShipFlow must not create or refresh a project-local `TASKS.md` symlink.
- If an older ShipFlow-created project-local `TASKS.md` symlink points into `shipglowz_data`, ShipFlow may remove that legacy symlink.
- A real app-owned project `TASKS.md` must be left untouched.

## Observed Behavior

- Earlier behavior created or repaired project-local `TASKS.md` symlinks that pointed into `shipglowz_data`.
- Broken symlinks caused `ln: ... File exists`; non-broken symlinks still polluted the app project with ShipFlow-owned tracking.
- The app still starts, but project-local tracker symlinks violate the intended ownership boundary.

## Evidence

- Operator log: `ln: failed to create symbolic link '/home/ubuntu/nococaine/TASKS.md': File exists`.
- Local inspection: `/home/ubuntu/nococaine/TASKS.md -> /home/claude/shipglowz_data/projects/quit-coke-app/TASKS.md`.
- Local inspection: `test -e /home/ubuntu/nococaine/TASKS.md` failed while `ls -l` showed the symlink path.
