# Bug Record: BUG-2026-05-06-001

## Summary

- Title: Stop Environment leaves orphaned PM2 apps and allows crash loops
- Current status: `closed`
- Severity: `high`
- Owner: unassigned
- Next command: `none`

## Reproduction

- Preconditions:
  - A PM2 app exists whose project directory has been deleted or moved.
  - The PM2 app was generated with automatic restart behavior.
  - PM2 startup is enabled through `pm2-claude.service`.
- Steps:
  1. Start a ShipFlow environment that can fail immediately.
  2. Delete or move the project path used by the PM2 app.
  3. Use ShipFlow Stop Environment or Stop All from the menu.
  4. Reboot or resurrect PM2 state.

## Expected Behavior

- Stop Environment should stop the selected PM2 app even when the project path is no longer resolvable.
- Stop All should include PM2 apps that are no longer discoverable from `.flox` project directories.
- A broken dev command should not be allowed to restart indefinitely and fill PM2 logs.
- PM2 state saved by ShipFlow stop operations should reflect the stopped state.

## Observed Behavior

- Stop Environment resolved projects from disk before stopping PM2, so orphaned PM2 apps could remain outside the normal stop path.
- Stop All iterated only disk-discovered Flox environments, not all PM2 entries.
- Non-Expo generated ecosystem configs used unbounded `autorestart: true`.
- A broken `contentflow_lab` entry accumulated more than 56 million restarts and multi-gigabyte logs.

## Evidence

- Runtime snapshot: `contentflow_lab` restart count `56402570`.
- Runtime snapshot: `/home/claude/.pm2/pm2.log` approximately `17.6G`.
- Runtime snapshot: `/home/claude/.pm2/logs/contentflow-lab-error.log` approximately `6.0G`.
- Code inspection: `lib.sh::env_stop`, `batch_stop_all`, and generated PM2 ecosystem config.
