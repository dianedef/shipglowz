# Bug Record: BUG-2026-05-04-002

## Summary

- Title: ShipFlow menu Back/cancel navigation has inconsistent feedback
- Current status: `closed`
- Severity: `medium`
- Owner: unassigned
- Next command: none

## Reproduction

- Preconditions:
  - ShipFlow CLI is installed or run from the repository.
- The operator wants to open the Updates menu from the top-level shortcut.
- Steps:
  1. Run `sf "u"` / `./shipflow.sh u`, or open `sf` and press `u` in the top-level menu.
  2. Observe whether the Updates action opens.
  3. Open the interactive root menu, press `d` for Dashboard or `o` for Stop All, and observe where the action output starts.

## Expected Behavior

- ShipFlow should treat `u` as the same top-level shortcut shown in the interactive menu.
- Top-level menu actions should start on a clean screen after `Your choice:`.
- Slow actions should show immediate feedback before doing longer checks.
- Back options should return directly to the parent menu without another pause prompt.
- `Esc` and `Backspace` should behave like `x` where a Back option exists.
- The Updates screen should open and keep the existing explicit confirmation before running package upgrades.
- Unknown shortcut arguments should fail visibly instead of falling into an ignored interactive loop.

## Observed Behavior

- `shipflow.sh` ignored command-line arguments because it called `main` without `$@`.
- `main` always entered `run_menu`, so `sf u` did not dispatch `action_updates`.
- In a non-interactive smoke run, the menu loop repeatedly attempted to read from `/dev/tty` and timed out.
- In the interactive fallback menu, action output could start on the same line as `Your choice:` because `menu_bash.sh` read keys directly and did not print a newline.
- Pressing `u` could look frozen because `updates_menu` refreshed apt/npm/pip/rustup counts before printing any feedback.
- Pressing `x` inside Updates returned from `updates_menu`, then the parent top-level menu wrapper still showed `Appuie sur une touche pour continuer...`.
- Root menu actions such as Dashboard and Stop All could render below the full root menu, leaving unrelated menu text above the command output.
