# Bug Record: BUG-2026-05-04-001

## Summary

- Title: Disk cleanup under-warns critical root disk pressure
- Current status: `closed`
- Severity: `high`
- Owner: unassigned
- Next command: none

## Reproduction

- Preconditions:
  - ShipFlow local CLI is running on a VM with the root filesystem near exhaustion.
  - Root disk pressure is high or critical, for example `/` at 90-95%+ used or only a few GiB free.
- Steps:
  1. Open ShipFlow.
  2. Run Cleanup from the menu.
  3. Choose light or aggressive cleanup.

## Expected Behavior

- ShipFlow should make disk pressure severity obvious before destructive cleanup choices.
- At critical pressure, the operator should see urgent language that names the VM freeze/build-stall risk and recommends immediate cleanup or disk expansion.
- After cleanup, ShipFlow should say whether disk pressure was cleared or remains risky.

## Observed Behavior

- The cleanup flow showed routine free-space and deletion-confirmation text.
- The menu could show a generic low disk warning, but it did not distinguish warning/high/critical states.
- In a real incident, `/` was at 94-95% used and the operator did not receive an urgent-enough warning before running aggressive cleanup.

## Evidence

- `lib.sh` cleanup flow before fix: `Free space before:` plus routine cleanup confirmation.
- Local validation after fix:
  - `bash -n /home/ubuntu/shipflow/lib.sh`
  - `disk_pressure_level` current state: `ok` at `74%` used and `12G` free.
  - Simulated states: `96%` -> `critical`, `91%` -> `high`, `86%` -> `warning`.
