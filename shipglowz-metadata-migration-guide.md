# ShipGlowz Metadata Migration Guide

## Purpose

This guide explains how to adopt ShipGlowz metadata and versioning in an existing project without creating regressions, duplicate sources of truth, or broken runtime content schemas.

Use it when:

- a project already has legacy root ShipGlowz docs such as `BUSINESS.md`, `BRANDING.md`, `GUIDELINES.md`, `CONTEXT.md`, `CONTENT_MAP.md`, specs, or research docs that must be moved into the canonical `shipglowz_data/` layout
- old docs need traceability, versioning, and dependency references
- a team wants to make ShipGlowz artifacts machine-checkable with `tools/shipglowz_metadata_lint.py`

Official migration scope for legacy adoption. These root names are migration sources only; they are not compliant final locations except `AGENT.md` and an explicitly adopted `CLAUDE.md`:

- `AGENT.md` when it is the active agent entrypoint
- `CONTEXT.md` and promoted specialized context docs when they are part of the official project context layer, migrated to `shipglowz_data/technical/`
- `CLAUDE.md` only when the project explicitly promotes it as an active repository-guidance artifact
- `BUSINESS.md`, migrated to `shipglowz_data/business/business.md`
- `PRODUCT.md`, migrated to `shipglowz_data/business/product.md`
- `BRANDING.md`, migrated to `shipglowz_data/branding/branding.md`
- `GTM.md`, migrated to `shipglowz_data/business/gtm.md`
- `ARCHITECTURE.md`, migrated to `shipglowz_data/technical/architecture.md`
- `GUIDELINES.md`, migrated to `shipglowz_data/technical/guidelines.md`
- `specs/*.md`, migrated to `shipglowz_data/workflow/specs/`

Everything else should be treated as case-by-case legacy material unless it is already part of the active ShipGlowz artifact workflow for that project.

## Canonical location rule

ShipGlowz now uses a simple split:

- root-level project files are limited to `README.md`, `AGENT.md`, `AGENTS.md` as a symlink to `AGENT.md`, optional `CLAUDE.md`, and project/tool-native docs that are not ShipGlowz governance artifacts
- `shipglowz_data/` inside each project is the canonical home for ShipGlowz governance artifacts
- `${SHIPFLOW_DATA_DIR:-$HOME/shipglowz_data}` is a legacy external path retained for historical or compatibility inputs only; active project truth is the local `shipglowz_data` corpus.

That means:

- move business, product, brand, GTM, architecture, technical context, guidelines, specs, research reports, review reports, verification reports, and decision records under the project's `shipglowz_data/` tree
- keep `TASKS.md` and `AUDIT_LOG.md` as operational trackers, preferably under `shipglowz_data/workflow/`
- `PROJECTS.md` is not required as active project governance; treat it only as a legacy migration artifact unless a project explicitly declares an external compatibility registry.
- do not keep duplicate root copies once the canonical `shipglowz_data/` artifact exists

If the external cross-project `shipglowz_data` needs visibility, reference the project-local artifact. Do not create a second canonical copy.

## What gets metadata

Metadata is required for ShipGlowz decision artifacts, including:

- `shipglowz_data/business/business.md`
- `shipglowz_data/business/product.md`
- `shipglowz_data/branding/branding.md`
- `shipglowz_data/business/gtm.md`
- `shipglowz_data/technical/architecture.md`
- `shipglowz_data/technical/guidelines.md`
- files in `shipglowz_data/workflow/specs/`
- research reports
- review reports
- readiness reports
- verification reports
- audit reports
- decision records
- architecture notes
- technical module context docs
- project docs generated by ShipGlowz that drive implementation or product decisions
- `CLAUDE.md` when the repo officially uses it as a maintained agent/repository contract rather than an informal tool note

For legacy bulk migration, keep the official default scope narrower:

- active context docs (`AGENT.md`, root `CONTEXT.md` and promoted `CONTEXT-*` docs only as migration sources)
- root `BUSINESS.md`, `PRODUCT.md`, `BRANDING.md`, `GTM.md`, `ARCHITECTURE.md`, `GUIDELINES.md` only as migration sources
- root `specs/*.md` only as migration sources

`CLAUDE.md` stays out of the bulk-default scope unless the repo has already decided it is part of the active official ShipGlowz artifact layer. Promote it intentionally; do not infer that from filename alone.

Do not keep widening the migration just because old docs exist. If a legacy file is not part of the active official ShipGlowz doc set for that project, leave it alone until there is an explicit reason to formalize it.

Metadata is not required for fast operational trackers:

- `TASKS.md`
- `AUDIT_LOG.md`
- `PROJECTS.md`

If a tracker contains a durable business or technical decision, extract that decision into a dedicated artifact instead of forcing frontmatter into the tracker.

## What must not be migrated blindly

Do not apply ShipGlowz frontmatter mechanically to:

- application runtime content such as Astro collections, MDX blog posts, CMS content, or app-rendered docs
- framework-specific docs that require their own schema
- user-facing content files whose frontmatter is parsed by the application

For those files, preserve the application schema first. ShipGlowz metadata only belongs where it will not break the runtime contract.

## Safe migration strategy

The default strategy is additive and conservative.

1. Identify explicit decision artifacts.
2. Leave operational trackers alone.
3. Preserve the existing markdown body.
4. Add only ShipGlowz frontmatter.
5. Use cautious values when information is uncertain.
6. Validate with the linter.
7. Upgrade versions and statuses later when real review happens.

This avoids rewriting content before the team has reviewed it.

## Minimal migration procedure

### 1. Inventory the docs

Classify each file as one of:

- decision artifact
- runtime content
- tracker / registry
- ambiguous historical note

If the file is ambiguous, do not force migration on the first pass.

### 2. Migrate only explicit artifacts first

Good first-pass candidates:

- `BUSINESS.md`
- `BRANDING.md`
- `GUIDELINES.md`
- specs in `specs/`
- existing research or audit markdown files with stable intent

For a large multi-project migration, this should usually become the default first wave:

- move every repo-level `BUSINESS.md`, `BRANDING.md`, and `GUIDELINES.md` to canonical `shipglowz_data/` paths with metadata
- move every root markdown file under `specs/` to `shipglowz_data/workflow/specs/`
- leave everything else for a second classification pass

This is the highest-value / lowest-risk slice because these files are explicit decision artifacts and are already part of the ShipGlowz linter default scope.

In most projects, this first wave is also the intended stopping point for legacy migration. Older audits, ad hoc research notes, historical plans, and one-off docs can remain outside ShipGlowz metadata unless they are promoted into the active official documentation set.

### 3. Add minimal frontmatter

Use the smallest honest set of fields:

```yaml
---
artifact: business_context
metadata_schema_version: "1.0"
artifact_version: "0.1.0"
project: "[project name]"
created: "2026-04-25"
updated: "2026-04-25"
status: draft
source_skill: unknown
scope: business
owner: unknown
confidence: low
risk_level: medium
security_impact: unknown
docs_impact: yes
linked_systems: []
depends_on: []
supersedes: []
evidence: []
next_review: "unknown"
next_step: "/sg-docs audit BUSINESS.md"
---
```

For a migrated legacy file:

- prefer `artifact_version: "0.1.0"` until it is reviewed
- prefer `status: draft` if the current correctness is not proven
- prefer `confidence: low` or `medium` over invented certainty
- use `unknown` when you do not know the answer

Artifact-specific conservative defaults are recommended:

- `BUSINESS.md` -> `shipglowz_data/business/business.md`, `artifact: business_context`
- `PRODUCT.md` -> `shipglowz_data/business/product.md`, `artifact: product_context`
- `BRANDING.md` -> `shipglowz_data/branding/branding.md`, `artifact: brand_context`
- `GTM.md` -> `shipglowz_data/business/gtm.md`, `artifact: gtm_context`
- project competitors/inspirations registry -> `artifact: competitive_intelligence`
- affiliate/referral/partner program registry -> `artifact: affiliate_program_registry`
- `ARCHITECTURE.md` -> `shipglowz_data/technical/architecture.md`, `artifact: architecture_context`
- `GUIDELINES.md` -> `shipglowz_data/technical/guidelines.md`, `artifact: technical_guidelines`
- `CLAUDE.md` -> `artifact: technical_guidelines`
- `specs/*.md` -> `shipglowz_data/workflow/specs/*.md`, `artifact: spec`
- `docs/technical/*.md` -> `artifact: technical_module_context`

When the document does not yet expose the missing metadata clearly, use placeholder-but-honest values that keep the artifact lintable without pretending certainty:

- `target_audience: unknown`
- `value_proposition: unknown`
- `business_model: unknown`
- `market: unknown`
- `brand_voice: unknown`
- `trust_posture: unknown`
- `linked_systems: []`
- `user_story: "unknown (legacy spec migrated to ShipGlowz metadata)"`
- `next_review: "unknown"`

### 4. Keep the body unchanged

Do not rewrite the document content during the frontmatter migration unless the task explicitly includes content refactoring.

The first pass is about traceability, not polishing.

This rule also means:

- do not re-template old documents just because templates now exist
- do not normalize headings during the metadata pass
- do not rewrite content to "improve" old docs unless that is a separate task

Templates are for future docs and deliberate refactors, not for bulk rewriting legacy material during migration.

### 5. Validate

Run:

```bash
SHIPFLOW_ROOT="${SHIPFLOW_ROOT:-$HOME/shipglowz}"
"$SHIPGLOWZ_ROOT/tools/shipglowz_metadata_lint.py"
```

Or validate only a narrow scope:

```bash
SHIPFLOW_ROOT="${SHIPFLOW_ROOT:-$HOME/shipglowz}"
"$SHIPGLOWZ_ROOT/tools/shipglowz_metadata_lint.py" BUSINESS.md BRANDING.md specs/
```

For multi-project migrations, validate repo by repo or by an explicit list of touched paths. This keeps failures local and easier to fix.

The linter path is ShipGlowz-owned. Resolve it from `${SHIPGLOWZ_ROOT:-${SHIPFLOW_ROOT:-$HOME/shipglowz}}` even when the migration is being applied to another project repository.

If a file already has frontmatter but fails the linter:

- do not overwrite the whole header blindly
- patch only the missing or inconsistent fields
- if `status` is already `ready`, `reviewed`, or `active`, ensure `artifact_version >= 1.0.0`

In practice, legacy migrations often encounter a few partially structured files. Treat them as exceptions to normalize carefully, not as a reason to downgrade the whole migration strategy.

### 6. Review later

Once the migrated document has been reviewed for factual quality, business coherence, or technical correctness:

- raise `confidence`
- update `status`
- bump `artifact_version`
- add stronger `evidence`
- add `depends_on` if the artifact now relies on other versioned contracts

## Versioning rules

ShipGlowz uses two separate version concepts.

### `metadata_schema_version`

This tracks the ShipGlowz metadata contract itself.

Change it only when the metadata structure or required fields change across the framework.

In normal project work, this usually stays stable.

### `artifact_version`

This tracks the meaning of the document.

Recommended convention:

- `0.x.y` for drafts, inferred migrations, and unreviewed legacy artifacts
- `1.0.0` for the first reviewed version that is safe to depend on
- patch bump for wording or evidence updates that do not change the decision
- minor bump for changed assumptions, audience, scope, or behavioral expectations
- major bump for incompatible decision changes

Examples:

- `0.1.0 -> 0.1.1`: better wording, same decision, still unreviewed
- `0.1.0 -> 1.0.0`: first real review completed
- `1.0.0 -> 1.1.0`: target audience, feature scope, pricing assumption, or docs obligations changed
- `1.2.0 -> 2.0.0`: incompatible architecture direction, business model, public promise, or security posture changed

## Status rules

Use status to express trust, not to decorate the file.

- `draft`: not yet reviewed enough to be a strong dependency
- `reviewed`: reviewed and acceptable to depend on
- `active`: current governing contract in active use
- `stale`: historically useful but no longer aligned with the product
- `superseded`: replaced by a newer artifact

Recommended coupling:

- `0.x.y` pairs naturally with `draft`
- `>= 1.0.0` should usually mean `reviewed` or `active`

Do not mark a migrated legacy artifact `reviewed` unless someone actually reviewed it.

## Dependency tracking

Use `depends_on` when one artifact is based on another versioned contract.

Example:

```yaml
depends_on:
  - artifact: BUSINESS.md
    artifact_version: "1.0.0"
    required_status: reviewed
  - artifact: BRANDING.md
    artifact_version: "1.1.0"
    required_status: reviewed
```

This is useful when:

- a spec is written against a given business contract
- a verification report depends on a reviewed spec
- a marketing page depends on a current brand contract

If the dependency changes later, ShipGlowz can flag potential drift instead of relying on memory.

## Templates

When a file needs deeper restructuring, use the skill-aligned templates in:

```text
templates/artifacts/
```

Do not use templates to overwrite a healthy existing document structure just because the headings are different.

The right order is:

1. preserve valid legacy structure
2. add metadata
3. lint
4. refactor content structure only if the skill workflow benefits from it

## Recommended migration sequence for an existing project

1. Move root `BUSINESS.md`, `BRANDING.md`, and `GUIDELINES.md` to canonical `shipglowz_data/` paths.
2. Move stable root docs in `specs/` to `shipglowz_data/workflow/specs/`.
3. Migrate active project competitor/inspiration and affiliate/referral/partner registries when they govern positioning, monetization, public recommendations, or disclosure.
4. Migrate research, review, readiness, verification, and audit artifacts.
5. Leave trackers untouched.
6. Review runtime content separately and only if ShipGlowz metadata can coexist safely.

This sequence minimizes risk and gives the highest coordination value first.

## Multi-project rule

For a workspace with many repos, do not start from `shipglowz_data` unless it clearly hosts canonical decision artifacts.

The practical order is:

1. migrate canonical docs in each project repo
2. keep `shipglowz_data` limited to tracking and registry roles
3. avoid editing both a project copy and a `shipglowz_data` copy of the same artifact

If a duplicate exists, choose the project repo as canonical for business and technical decision docs, then migrate only that copy.

## Stopping rule

A legacy migration should be considered complete when:

- `BUSINESS.md`, `BRANDING.md`, `GUIDELINES.md`, and `specs/*.md` are migrated in the target repos
- tracker files remain excluded
- runtime content remains untouched
- old ad hoc docs outside the official ShipGlowz doc set are explicitly left out of scope

Do not turn migration into an endless cleanup of historical markdown. If the file is not part of the official ShipGlowz artifact set, leave it for a future case-by-case decision.

## Definition of done for migration

A project migration is complete when:

- explicit decision artifacts have ShipGlowz frontmatter
- trackers remain excluded
- runtime content schemas are not broken
- canonical artifact location is clear
- `$SHIPGLOWZ_ROOT/tools/shipglowz_metadata_lint.py` passes on the intended scope
- version and status values are honest

## Anti-patterns

Avoid these mistakes:

- adding metadata to `TASKS.md` just for consistency
- migrating both a project copy and a `shipglowz_data` copy of the same decision doc
- declaring `1.0.0` without review
- marking artifacts `reviewed` because they were migrated
- overwriting a useful legacy doc body with a generic template
- forcing ShipGlowz frontmatter into app runtime content that expects another schema

## Quick checklist

- Is this file a decision artifact?
- Is this file canonically stored in the project repo?
- Can ShipGlowz metadata be added without breaking runtime parsing?
- Did I preserve the body?
- Did I choose conservative version/status/confidence values?
- Did I run the linter?

If all answers are yes, the migration is probably safe.
