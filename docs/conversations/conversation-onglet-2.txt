• Oui, ton intuition est bonne. Le workflow actuel expose trop
  l’utilisateur à la mécanique interne. Pour un utilisateur final
  non développeur, spec -> ready -> spec -> ready -> start ->
  verify -> test ne devrait pas être une série d’actions
  manuelles. Ça devrait être un moteur interne.

  Je ne mettrais pas cette boucle directement dans sf-spec. Je
  garderais sf-spec, sf-ready, sf-start, sf-verify, sf-test comme
  briques atomiques, et je créerais une skill maître, par exemple
  sf-do, sf-deliver ou sf-build, qui orchestre tout.

  Architecture idéale :

  Utilisateur
     │
     ▼
  /sg-do "Je veux que..."
     │
     ├─ explore rapide si besoin
     │
     ├─ spec draft
     │
     ├─ ready review adverse
     │
     ├─ spec repair
     │
     ├─ ready review
     │      ↺ boucle max 2-4 fois
     │
     ├─ si bloqué : poser UNE question utile à l’utilisateur
     │
     ├─ start / implémentation
     │
     ├─ verify
     │
     ├─ check/test
     │
     └─ rapport final court

  Le point important : l’utilisateur ne devrait pas voir les
  rapports intermédiaires sauf si une décision produit est
  nécessaire. Les rapports détaillés peuvent rester dans les
  fichiers de spec, logs ou chantier history, mais la conversation
  devrait dire seulement :

  Je cadre le besoin.
  Je vérifie que la spec est solide.
  J’implémente.
  Je vérifie.
  Terminé : voici ce qui a changé et ce qu’il reste à savoir.

  À mon avis, il faut donc deux niveaux de skills :

  NIVEAU UTILISATEUR FINAL
  ────────────────────────
  sf-do / sf-deliver
  - prend une story floue
  - infère les décisions raisonnables
  - pose seulement les questions bloquantes
  - fait le travail de bout en bout
  - donne un rapport court

  NIVEAU INTERNE AGENT
  ────────────────────
  sf-explore
  sf-spec
  sf-ready
  sf-start
  sf-verify
  sf-test
  sf-check
  sf-ship

  Je ne rendrais pas sf-spec auto-récursive par défaut, parce que
  ça mélange deux responsabilités : écrire une spec et piloter un
  workflow. Par contre, sf-spec peut devenir compatible avec une
  orchestration automatique : sortie structurée, statut clair,
  liste exacte des gaps, mode “silent/internal”, etc.

  La boucle devrait être dans la master skill avec des garde-
  fous :

  while spec not ready:
    run spec-writer
    run ready-reviewer
    if ready:
      continue
    if gaps are answerable by code/context:
      run spec-repair
    if gaps need product/security decision:
      ask user one concise question
    if iterations > max:
      stop with short blocker report

  Je mettrais une limite stricte, genre 3 passes. Sinon l’agent
  peut tourner en rond en améliorant la spec indéfiniment.

  Le vrai changement de philosophie serait celui-ci :

  Aujourd’hui :
  Utilisateur pilote le workflow.
  Agent exécute chaque étape.

  Demain :
  Utilisateur donne l’intention.
  Agent pilote le workflow.
  Utilisateur intervient seulement pour les décisions métier.

  Donc oui : je créerais une master skill. Probablement pas sf-
  start, parce que sf-start est déjà “exécution depuis contrat”.
  Je créerais plutôt une nouvelle skill user-facing, par exemple :

  sf-do
  Args: story utilisateur
  But: livrer le résultat de bout en bout avec orchestration
  autonome.

  Et dans ses règles :

  - rapports intermédiaires courts, pas de détails techniques par
  défaut
  - sous-agents autorisés pour spec, ready, implementation, verify
  quand le scope le justifie
  - questions utilisateur uniquement si l’absence de réponse
  change le produit, la sécurité, les permissions, les données, le
  coût, ou un comportement irréversible
  - inférence par défaut pour le reste
  - rapport final en 5-10 lignes maximum, avec détails techniques
  seulement sur demande

  Ça correspond beaucoup mieux à ton public cible. Les skills
  atomiques restent utiles pour nous, mais elles deviennent une
  plomberie interne, pas l’interface principale.

› oublie pas les accents en français. est ce que mettre les skills en
  anglais serait mieu pour l'IA ?


• Oui, je garde les accents pour le français désormais, sauf dans les
  clés techniques où l’ASCII est préférable.

  Pour les skills : je ne mettrais pas tout en anglais.

  Le meilleur compromis pour ShipFlow, c’est :

  - Commandes / noms de skills : anglais court
    sf-build, sf-ready, sf-verify, sf-ship
  - Schéma machine / headings stables : anglais
    Status, User Story, Acceptance Criteria, Skill Run History
  - Questions utilisateur / rapports / explications : français avec
    accents
    C’est ton interface produit. L’utilisateur final doit comprendre
    vite, sans jargon.
  - Instructions internes très critiques : français ou anglais, mais
    ultra structurées
    Les modèles comprennent très bien le français. Ce qui compte le
    plus, c’est la précision : must, never, stop conditions, listes,
    critères vérifiables.

  Donc ma reco :

  Interface utilisateur : français
  Contrats techniques : anglais stable
  Instructions de workflow : français structuré
  Commandes : anglais court

  Traduire toutes les skills en anglais pourrait légèrement aider
  certains modèles sur des patterns techniques, mais ça ferait perdre
  en maintenabilité pour toi et en clarté produit. Le vrai gain IA
  vient surtout de la structure, pas de la langue.

  Pour sf-build, je garderais donc les prompts utilisateur en français
  impeccable, avec options préparées, et les sections internes
  compatibles avec les specs actuelles.


› Comment ça, instruction de workflow, ça veut dire quoi?


• Par “instructions de workflow”, je parle du texte dans les SKILL.md
  qui dit à l’agent comment dérouler une tâche.

  Exemples :

  ### Step 1 — Comprendre le besoin
  - Lire la spec existante si elle existe.
  - Si une décision produit reste ambiguë, poser une question.
  - Ne pas coder tant que le scope n’est pas validé.

  Ou :

  ### Step 4 — Vérification
  - Lancer les tests ciblés.
  - Vérifier que la user story est réellement tenue.
  - Si la vérification échoue, ne pas lancer sf-ship.

  Donc ce n’est pas l’interface utilisateur, ni les noms de commandes.
  C’est la “notice interne” de la skill : les règles, les étapes, les
  garde-fous, les conditions d’arrêt.

  Dans ShipFlow, ça correspond surtout à :

  - les Step 1, Step 2, etc.
  - les Rules
  - les Guardrails
  - les Stop conditions
  - les critères “ready / not ready”
  - les consignes du type “si X arrive, alors lancer Y ou demander à
    l’utilisateur”
